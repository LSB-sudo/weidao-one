# 串口3蓝牙接口说明

## 当前目标

当前串口 3 的目标不是承载复杂协议，而是先确保：

- 蓝牙模块通过 `USART3` 的基础收发正常
- 可以快速验证“发什么，回什么”

因此本轮已将串口 3 清理为最小可测接口。

## 当前实现

相关文件：

- `MiniBalance_HARDWARE/USART3/usart3.c`
- `MiniBalance_HARDWARE/USART3/usart3.h`

当前保留内容：

- `uart3_init()`：串口 3 基础初始化
- `USART3_SendByte()`：发送单字节
- `USART3_SendBuffer()`：发送缓冲区
- `USART3_IRQHandler()`：接收中断 + 原样回显

## 当前测试逻辑

当前 `USART3_IRQHandler()` 中已经加入：

- 接收到什么字节
- 就立刻通过 `USART3_SendByte()` 回显什么字节

也就是：

- 蓝牙端发 `A`
- 串口 3 就回 `A`

用于验证：

- RX 正常
- TX 正常
- 中断接收路径正常

## 本次移除内容

本轮已移除串口 3 中与当前测试目标无关的旧逻辑，包括：

- 帧头/帧尾协议解析
- 校验和判断
- 通过串口 3 直接修改 `Target_EncoderC`
- 通过串口 3 直接修改 `Target_EncoderD`

这些逻辑会让串口 3 和控制业务耦合，不利于先确认蓝牙基础收发是否正常。

## 当前建议

先只做最简单测试：

1. 蓝牙模块连接成功
2. 发送任意字符/字符串
3. 检查是否完整原样回显

若这一步稳定，再决定后续是否在串口 3 上重新增加上层协议。

## 后续扩展建议

如果后续要把串口 3 继续往上做，建议顺序是：

1. 先保留当前回显能力作为基础连通性测试
2. 再在 app 层增加命令处理
3. 不建议再把业务协议直接塞回 `USART3_IRQHandler()`

更合理的方向是：

- 硬件层 `USART3` 只负责收发
- app 层负责命令解析与业务处理
