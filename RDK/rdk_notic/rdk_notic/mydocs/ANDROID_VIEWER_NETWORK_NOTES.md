# Android Viewer 网络与实现方案说明

## 1. 当前实现概览

- Android App 第一版采用 WebView 壳形式，当前默认加载 http://192.168.3.142:8080/viewer。
- RDK X5 端提供基于 WebRTC 的 /viewer 页面，以及对应的信令接口。
- 现阶段该方案更适合同一局域网内的快速联调与现场演示。

## 2. 是否必须同一个局域网

- 当前实现默认要求 Android 手机能够直接访问 RDK 的局域网 IP 地址。
- 从技术上看，并不是永远必须处于同一个局域网；如果要跨网络访问，需要额外引入网络穿透或中继方案。
- 可选的跨网方案包括：VPN 或组网工具（如 Tailscale、ZeroTier）、TURN 服务、SFU 架构、云端中继等。
- 对评委现场展示而言，建议保持手机与 RDK 处于同一局域网，风险最低，排障最直接。

## 3. 当前 WebView 实现弱点

- 依赖 Android System WebView 或 Chrome 对 WebRTC 的支持能力，兼容性和行为受系统环境影响。
- App 对 WebRTC 状态机、重连过程、错误处理的控制能力有限。
- 当前 UI 本质仍然是网页，原生交互和原生体验有限。
- 当前访问链路使用明文 HTTP，不适合直接面向更复杂或更开放的部署环境。
- 当前访问地址中 IP 写死，局域网 IP 变化后需要同步调整。
- 目前尚未具备鉴权、多用户能力、云端转发等产品化特性。
- 当前板端编码链路仍然是 ffmpeg/libx264 的 PoC 方案，不是最终硬编优化版本。

## 4. 是否需要原生 WebRTC

- 面向当前评委展示阶段，暂时不需要把 Android 端改造成原生 WebRTC 客户端。
- 如果后续要走产品化或长期稳定版本，建议评估接入原生 WebRTC SDK。
- 原生 WebRTC 的主要优点包括：更好的状态控制、更完善的重连机制、更自然的原生 UI、更可控的权限管理与后台行为，以及更高的整体稳定性。
- 对应代价是开发量明显更大，联调复杂度和维护成本也会提升。

## 5. RDK 换局域网 IP 会不会变

- 如果当前网络环境使用 DHCP 自动分配地址，RDK 的局域网 IP 大概率会变化。
- 一旦地址变化，当前 App 中写死的 192.168.3.142 将直接失效。

## 6. 推荐解决方案路线

### 近期 / 展示阶段

- 在路由器侧对 RDK 做 DHCP 绑定，尽量固定局域网 IP。
- Android App 增加 IP 设置页，允许手动修改访问地址。

### 现场独立演示

- 可考虑由 RDK 自身开启热点，让手机直接连接 RDK 热点完成演示，减少外部网络变量。

### 辅助方案

- 可尝试使用 mDNS，例如主机名 rdk-x5.local，降低对数字 IP 的依赖。
- 但需要注意 Android 端与 WebView 对 mDNS 的兼容性并不总是稳定，不能作为唯一依赖。
- 可在板端或配套界面中展示二维码，便于手机快速扫码打开当前访问地址。

### 跨网 / 远程访问

- 若后续存在远程访问需求，可再评估 Tailscale、ZeroTier、TURN、SFU、云端中继等方案。

## 7. 下一步建议

- 优先给 Android App 增加 IP 设置页，并将配置保存到 SharedPreferences。
- 支持用户输入单独的 IP，或直接输入完整 URL。
- 默认值继续保留为 http://192.168.3.142:8080/viewer，兼顾当前演示路径与后续可配置性。

### RDK -> 32 -> 蓝牙 -> App 的 IP 同步方案

- 背景：RDK X5 后续会通过串口连接 32；32 有蓝牙接口，蓝牙链路相对稳定，不依赖 RDK 的局域网 IP。
- 思路：RDK 启动时读取自身当前局域网 IP，通过串口发送给 32；当 IP 变化时再次发送；32 通过蓝牙把最新 IP 上传给 App。
- App 侧：App 收到蓝牙 IP 后自动更新 viewer URL，例如 `http://<ip>:8080/viewer`，无需用户手动输入。
- 触发时机：RDK 服务启动、网络连接变化、定时检测发现 IP 变化。
- 建议消息格式：简单文本或 JSON，例如：
  - 文本：`RDK_IP=192.168.43.12;PORT=8080\n`
  - JSON：`{"type":"rdk_ip","ip":"192.168.43.12","port":8080}`
- 风险/注意事项：
  - RDK 可能有多个网卡/IP，需要选择手机可访问的那一个。
  - 蓝牙传输只解决“发现地址”，视频仍走 Wi-Fi/WebRTC。
  - 需要避免频繁重复上报，可只在 IP 变化时上报。
  - App 应保留手动输入 IP 的兜底方式。
- 推荐定位：这是比 mDNS 更可控的现场展示地址发现方案，适合当前系统，因为 32 蓝牙链路稳定。
